PRACTICE MANAGER INTERFACE

TRANALYSIS is, contents-wise, directly modelled on NOMACCS. But unlike NOMACCS which is �descended� from NOMCODES, the underlying �code� of a TRANALYSIS record has been preset by our software, so it does not link to NOMCODES. Also, with the one important exception of code PM09 (Profit Costs Paid), the preset analysis codes do not hold accounting data but instead hold Time Recording and other practice data.

You can see a list of TRANALYSIS codes within Practice Manager in any of the following places:

File -> Setup -> Balance Loading -> Analysis Budgets

File -> Setup -> Balance Loading -> Analysis Balances

Reports -> Business Reports -> Management Report: Drop down lists in the Code Selection control

Reports -> Business Reports -> Departmental Report: Drop down list in the Column Content panel


Calculating Period End balances

Just as with NOMACCS, note that the CLOSING-BAL and CLOSING-BUDG fields always contain cumulative actual or budget balances to date, but not to any given historic period end. For example if it is now Period 7 there is no figure directly available in the table for a balance as at the end of Period 4. To report on historic information you will therefore have to create formulae to calculate these intermediate cumulative balances.

For example to derive the end of Period 4 actual balance you can create a formula called, say, YTDPeriod4 consisting of the following expression:

TRANALYSIS.OPENING-BAL + TRANALYSIS.PERIOD-BAL@1 + TRANALYSIS.PERIOD-BAL@2 + TRANALYSIS.PERIOD-BAL@3 + TRANALYSIS.PERIOD-BAL@4

FEE-EARNER and MT-TYPE will always be used - unlike NOMACCS where you have control over whether or not postings are analysed by fee earner and matter type.

Second category of TRANALYSIS codes - user defined

The second category is derived from just those Time Recording Activity Codes where you have set the Analysis flag to Yes. It follows that the number of these codes on your system can be anything from zero upwards, limited only by the number of activity codes you have chosen to create and mark for Analysis.

The actual format of analysis codes in this category comprises the Activity code itself, followed by the same set of suffixes as the TIME codes shown above, from A through to P. Note that 3 of the 16 TIME codes ie those with suffixes J K and L are exclusive to non-chargeable time recording, so chargeable activity codes will have a maximum possible 13 analysis codes per fee earner/matter type. Non-chargeable activity codes will have a maximum possible three analysis codes viz those suffixed J K or L.

For example suppose you have an activity code of PTEL for telephone calls on private client work, the effect of switching on its analysis flag will be to create and populate no less than 5 new TRANALYSIS records as soon as the very first time posting of this activity is recorded viz:

PTELACost of time spent on the phone
PTELBChargeable value of time spent on the phone
PTELDWIP at cost of time spent on the phone
PTELEWIP at chargeable value of time spent on the phone
PTELHMinutes if Chargeable �standard time� spent on the phone

Other TRANALYSIS codes starting with PTEL will be automatically created by the software during the year if and when billing and closure and write offs take place. Also remember that a new set of PTEL records is created for each new Fee earner and Matter Type combination that engages in posting PTEL time entries, so everyone using the PTEL time posting activity code is going to generate a minimum of five more TRANALYSIS records!

Thus TRANALYSIS codes starting with TIME give you a picture of time and WIP per fee earner/matter type but without reference to any particular activity, while TRANALYSIS codes starting with your own Time Activity codes give the same fee earner/ matter type breakdown but by reference to activities.

There are a possible maximum 13 heads of TIME analysis for chargeable work, so any one of your time activity codes could spawn no less than 13 TRANALYSIS records for each fee earner/matter type combination engaged in that activity. A fresh set of TRANALYSIS records is then generated by the software for each new financial year of your system�s operation. You will by now have realised this could quickly generate an enormous number of TRANALYSIS records, so we recommend you don�t flag individual activity codes for Analysis without careful thought. Users are more likely to switch analysis on for Non chargeable and leave it off for Chargeable activities but this isn�t a fixed rule and your firm may have reasons to deviate from it.

Note in this context that future amendments to the Activity record have no retrospective effect, so switching on the Activity�s analysis flag won�t generate TRANALYSIS data for previous time entries for that activity. Similarly, switching off the Activity�s analysis flag won�t destroy existing TRANALYSIS records for that activity.

Optional formatting and conversion of different kinds of data

All budget and period figures are numbers but they represent a range of sometimes incompatible information viz:

Integer numbersPM01 to PM07
Integer money valuesPM10
Decimal money valuesPM09
 TIME A to TIME G
 TIME L to TIME P
 All Time Activity codes suffixed A to G and L to P
 Integer time in minutesTIME H to TIME K
 All Time Activity codes suffixed H to K

Be careful not to create data columns or formulae which purport to add or subtract these wholly different kinds of number to or from one another!

When reporting on codes that are integer numbers eg a report showing the number of new matters opened, take advantage of Crystal�s field formatting ie right click on these fields and then format them so that they will show simply as integers rather than with an annoying and repetitive .00 after each figure.

When reporting on codes that are integer time in minutes you can convert them for presentation purposes to hours and minutes using a formula, as in the following examples:


This one shows details for period 3:

ToText(Truncate({TRANALYSIS.PERIOD-BAL@3}/60),"####0")+ " hrs "
+
ToText(Remainder({TRANALYSIS.PERIOD-BAL@3},60),"00")+ " mins"


This one gives you Fee Earner totals for period 3 to place in a Fee Earner group header or footer:


ToText(Truncate(Sum({TRANALYSIS.PERIOD-BAL@3},{TRANALYSIS.FEE-EARNER})/60),"####0")+ " hrs "
+
ToText(Remainder(Sum({TRANALYSIS.PERIOD-BAL@3},{TRANALYSIS.FEE-EARNER}),60),"00")+ " mins"


LINKING AND APPLICATION

TRANALYSIS will be useful in a wide variety of management reports for summarising time recording, costs paid or practice statistical data. Each TRANALYSIS record is essentially self-contained - it holds its own text description � this contrasts with NOMACCS where you need NOMCODES to provide the account description. But it is not possible to �drill down� from TRANALYSIS into tables such as TRTRANS or NOMTRANS or CLIDB or MATDB. In other words it is not possible to verify the figures in TRANALYSIS against any of these tables by direct linkage. However, most time related totals can be derived independently of TRANALYSIS by reading and totalling selected detail held in TRTRANS, while the PM codes (apart from PM09) can ultimately be checked against opening and closing dates in tables like CLIDB and MATDB.

Also it is not possible from TRANALYSIS for you to simply drill down into the totals presented for Costs Paid in analysis code PM09 to establish the detail such as payment dates, individual client matter codes, bills delivered values and cash receipts per matter. However these can with difficulty be derived via the NOMTRANS and PAYMENTS tables - but unfortunately this can be complicated, and if at all possible it is much easier if you can simply rely on the summary Fee Earner and Matter Type period totals held in PM09 instead.

Although TRANALYSIS cannot be linked effectively to important transaction detail or database master tables like NOMTRANS, TRTRANS, CLIDB or MATDB it is OK to use its FEE-EARNER and MT-TYPE fields for linking up in the usual way with the FEETR, DEPARTMENT and MATTYPE tables respectively to provide full fee earner names, department names and matter type descriptions for grouping and presentational purposes.
