Interface Notes

All actual accounts transactions both on client matters and nominal accounts are recorded in NOMTRANS. The only accounting-type entry that is excluded from NOMTRANS is pencilled anticipated disbursements (either type AD, ADV or ADVU) because the posting values on pencilled anticipated disbursements can be amended up to the point of posting. Such pencilled items can instead be found in the SOLLEDGER table. Once posted these disbursements will be found in NOMTRANS in the usual way.

Creating a posting slip does not in itself generate a NOMTRANS record: instead a "provisional" record is created in TPTRANS. A NOMTRANS record is only created when the posting slip is later processed into an actual transaction.


Field Notes

Each posted transaction is represented in NOMTRANS by one or more separate transaction records. Certain posting types (e.g. CP, OP, OPV) are contained wholly within one record, and in these simple examples there is a nominal control code in the NL-CODE field and a different code representing a cashbook in the CONTRA-CODE field. These two codes represent the nominal accounts whose balances will be updated as a result of the transaction.

Note that many types of single posting in Practice Manager will generate multiple entry records in NOMTRANS behind the scenes. For example:

Client-to-Client Transfers (CC) generate two records

Client-to-Office Transfers (CO) generate just one record for the client-part of the transaction, but multiple records for the office-part. For example, a CO paying off three bills will generate four entries for the office-part, i.e. one for each of the three bills and one more for the entry which updates the office cashbook, making a total of five entries for the one CO actually posted.

Bill to Client (BC) or Credit Note (CN) postings generate one record for each fee earner with a share of profit costs in the bill or credit note, plus one record for the total of paid disbursements billed (ie disbursements that have X05 as their NOMTRANS.NL-CODE), plus another record for the total of anticipated disbursements billed (ie disbursements that have X15 as their NOMTRANS.NL-CODE).

Multi-line nominal journals generate one record for each nominal code involved in the transaction whether analysis heading or cashbook.

These multiple records all bear the same NOMTRANS.TRANS-NO but have different NOMTRANS.ENTRY-NO. The software gives the first entry a number of 1, the second a number 2 and so on. There should never be a jump in the number sequence of entries all bearing the same TRANS-NO.

CONTRA-CODE is not used at all for many posting types. Instead, successive entries of the same transaction each have one affected nominal code in the NL-CODE field. The sign of the NET-VALUE and VAT-VALUE fields signals whether the nominal balance is to be debited or credited. Here are examples where this field is not used:

  • Multi-line nominal entries
  • OR (Office Receipt) and AC (Allocate Credit) postings (matter office receipts or allocations of previous office receipts)
  • NCT (Nominal Cash Transfer) postings
  • NBT (Nominal Balance Transfer) postings

In a very simple example suppose you post an OR which is used wholly to pay off one outstanding bill to client. This will produce two NOMTRANS records with the same TRANS-NO. The first record will have the following key fields:

  • ENTRY-NO = 1
  • NL-CODE = Your office cashbook code
  • NET-VALUE = Amount received (positive amount)

The second record will have:

  • ENTRY-NO = 2
  • NL-CODE = "X20" (SOS present debtor control account)
  • NET-VALUE = Amount received (negative amount)

The VAT generated by a posting does not have its own separate NOMTRANS record but instead features in the VAT-VALUE and/or CASHVAT-VALUE fields of the net item it relates to. For example a NOMTRANS record for profit costs on a bill of £100 would show:

  • NET-VALUE = -100.00
  • NL-CODE = "X08"
  • CONTRA-CODE = "X20"
  • VAT-VALUE = -20.00
  • CASHVAT-VALUE = 0.00

The only exception to this is any nominal posting you make directly to the VAT control in which case the code VAT will feature in the NL-CODE or CONTRA-CODE fields in the usual way.


FEE-EARNER for multi-entry BC-type transactions is likely to be different for each entry, representing each different fee earners' share of costs delivered.

MT-TYPE is more usually the same throughout an entire BC transaction unless you have taken the non-default option to change matter types mid-bill.

BANKREC-STATUS contains codes downloaded from your bank when running the bank reconciliation program.

BILLED-PAID-FLAG may take one of the following values for bill postings, i.e. all NOMTRANS where POST-TYPE = "BC":

  • "W" = written off
  • "C" = credit noted

In other situations ie where the bill is either unpaid or has been settled by actual payment the field will be blank.

PENDING-FLAG is used for any transfer transactions entailing a movement between two cashbooks. If you Execute the transfer at posting time then an X will be written permanently into this field. If you Journal the transfer then a P for pending will be written into this field. Later on , when you run the Journal Transfer Update, this P will be replaced with an X.

CASHVAT-CODE is always the same as VAT-CODE.

CASHVAT-VALUE represents the VAT control movement that would be generated by the transaction if you accounted for VAT on a cash receipt / cash paid basis.

VAT-VALUE represents the corresponding movement for the same transaction if you account for VAT on a bills rendered / invoices received basis. Both this and CASHVAT-VALUE are updated automatically by the software.


Linking And Application

NOMTRANS provides the most detailed level of Practice Manager standard accounting reports. However, the table can be extremely large: it would not be unusual for a medium sized firm to have in excess of a million records in this table. To report without any record selection filter could be very slow. To boost performance the wide variety of indexes therefore eases the problem by facilitating faster reporting on small selections of data instead.

Detailed information on important things like outstanding bills and unbilled disbursements is all contained within NOMTRANS itself: simple links to CLIDB, MATDB, FEETR flesh out the report with client and fee earner names, or matter descriptions and ledger balances:

NOMTRANS.CL-CODE -> CLIDB.CL-CODE

NOMTRANS.MT-CODE -> MATDB.MT-CODE -> MATAC.MT-CODE

NOMTRANS.FEE-EARNER -> FEETR.FEE-EARNER


Other specialised tables will look up NOMTRANS:

SOLLEDGER.ACC-SEQ -> NOMTRANS.TRANS-NO

PURCHASES.TRANS-NO -> NOMTRANS.TRANS-NO

SOLLEDGER is a subset of NOMTRANS and keeps some of the details of Practice Manager’s “pencilled” anticipated disbursements. The link illustrated could serve to list those that have finally been posted. Further, if you change the link to Left Outer Join then you can also report on those that have not yet been posted.

PURCHASES keeps some of the supplier details of transactions posted as Creditor invoices to the nominal ledger or as posted (not merely pencilled) anticipated disbursements on client matters.


Aliased Links To NOMTRANS

NOMTRANS can be added to a report two or more times to see details of transactions that are connected to one another. For example anticipated disbursements will subsequently be billed, and the disbursements and billing transactions are of course quite distinct. This example would let you see details of anticipated disbursements as well as details of the later posting that included them in a bill:

SOLLEDGER.ACC-SEQ -> disbNOMTRANS.TRANS-NO

SOLLEDGER.BILL-TRANSNO -> billNOMTRANS.TRANS-NO
